Lead case study

Love That Gives Back

A full-stack fundraising platform for personal celebrations, built around a public guest donation flow and a private organiser dashboard.

Full-stack web app GitHub
<<<<<<< HEAD

Problem

Guests need a simple, trustworthy donation experience. Event hosts need visibility into donations without exposing payment logic or making public users create accounts just to contribute.

Approach

I structured the product as a public event experience plus an authenticated dashboard. Guests can browse an event page, choose a linked charity, and complete a donation without login, while hosts manage event details and review donation activity separately.

Architecture

React and Vite handle the public pages and dashboard routes. A Django REST backend manages event, charity, and donation APIs, with Stripe checkout and webhook handling controlling payment state transitions.

Stack

React 19, Vite, Django 5, Django REST Framework, Stripe, AWS Cognito, Redis, Celery, SQS, Lambda, SQL-backed data models, and automated tests across both frontend and backend.

Technical decisions

  • Kept the guest donation path account-free to reduce friction on a public payment journey.
  • Created Stripe sessions on the server so payment configuration and secrets stay off the client.
  • Used webhook-driven status updates and duplicate-processing guards for succeeded, failed, expired, and refunded donation states.
  • Added rate limiting, health checks, and operational logging around public-facing donation endpoints.

Outcome

The result is a real end-to-end web application with public traffic, private management workflows, payment processing, and transaction-aware state handling. The organiser dashboard surfaces event controls, raised totals, and donor counts rather than leaving the data buried in admin screens.

Why it matters for integrated engineering teams

This project demonstrates frontend and backend integration, API design, relational data handling, testing awareness, and careful treatment of user-facing transactions.

System flow

React routes Django REST API Stripe checkout and webhooks Dashboard and donation states

Evidence

  • Public event pages plus authenticated dashboard endpoints
  • Webhook tests for succeeded, expired, refunded, and replay-safe flows
  • More than 200 automated tests across frontend and backend files

Problem

Document-heavy drafting workflows are slow, difficult to trace, and easy to derail when requirements are missed or evidence is weak.

Approach

I broke the system into explicit stages: upload and parsing, requirement extraction, cited section generation, coverage analysis, unresolved evidence review, and export.

Architecture

The frontend is a Next.js workspace. The backend is a FastAPI pipeline with isolated services for parsing, validation, coverage, and export. Local development uses SQLite and filesystem storage, while the AWS deployment path is designed around RDS, S3, Bedrock, and Cognito-aware routing.

Stack

Next.js, React 19, FastAPI, SQLite, Postgres-ready data paths, S3, Amazon Bedrock, Cognito, Docker, GitHub Actions, and typed frontend utilities for traceability and quality diagnostics.

Technical decisions

  • Kept parsing, generation, coverage, and export as separate modules so failures are easier to diagnose and improve.
  • Added quality-signal panels and unresolved-gap views so users can see weak inputs before exporting output.
  • Built a cloud-ready deployment path with AWS readiness scripts covering ECS, RDS, S3, CloudFront routing, and config checks.
  • Protected delivery quality with backend tests, type checks, build verification, and docker smoke steps in CI.

Outcome

Nebula shows how I structure a system with clear boundaries, explicit API contracts, and a realistic deployment story rather than a single demo path. It is the clearest example in my work of combining product workflow, backend logic, and cloud-aware engineering decisions.

Systems-thinking proof of concept

Entropy

A proof-of-concept continual-learning knowledge engine that ingests information, generates hypotheses, critiques them, and updates a world model over time.

Proof-of-concept system GitHub

Problem

I wanted to explore continual learning as a software architecture problem, not just as prompt experimentation.

Approach

I designed a system with separate ingestion, memory, reasoning, orchestration, and API layers so each concern could be tested and evolved independently.

Architecture

FastAPI exposes ingestion and reasoning endpoints. The backend coordinates vector memory, graph memory, and episodic memory, while a React and Vite frontend acts as a lightweight dashboard over the system.

Stack

FastAPI, React, Vite, Qdrant, Neo4j, SQLite, Pydantic settings, Docker Compose, and pytest.

Technical decisions

  • Separated vector, graph, and episodic memory so each one has a clear role in retrieval, structure, and history.
  • Kept API endpoints narrow around ingestion, hypothesis generation, reasoning cycles, and graph inspection.
  • Used Docker Compose to stand up local supporting infrastructure instead of hiding dependencies behind assumptions.
  • Documented the architecture heavily so the proof-of-concept remains understandable as the system grows.

Outcome

Entropy is intentionally framed as a proof of concept, but it still shows disciplined modular design, real API surfaces, infrastructure awareness, and a test suite covering key behaviors.

Why it matters for integrated engineering teams

This work demonstrates curiosity, systems thinking, interface separation, and a willingness to decompose complex ideas into maintainable parts.

System flow

FastAPI API Reasoning loop Qdrant, Neo4j, SQLite React dashboard

Evidence

  • Documented ingestion to hypothesis to critique to update loop
  • Docker Compose stack for local Qdrant and Neo4j services
  • 30 pytest tests across API and reasoning modules
